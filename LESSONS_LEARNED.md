# Lessons Learned

## 2026-06-27 — 实现最小 LimdC 编译器和 UI 示例编译

### 经验

1. **示例驱动实现**: 仓库只有 `Example/newUI/main.lc` 时，以该文件作为第一验收用例可以快速建立编译链路。
2. **UI 自行转换**: 将 `window` 和 `button` DSL 转成 Win32 C++，再调用 `g++`，能避免让 C/C++ 编译器直接理解 DSL。
3. **保留生成代码**: 保留 `main.generated.cpp` 便于检查 include、窗口和按钮转换结果。
4. **本地临时目录**: g++ 在包含中文用户名的系统 Temp 路径下可能无法创建临时对象文件，设置项目内 `.limdc_tmp` 可绕开该问题。

### 教训

1. **不要假设系统 Temp 可用**: Windows 环境下工具链可能受用户名编码或临时目录影响，目标编译命令应控制临时目录。
2. **先实现最小语法闭环**: 当前不需要完整 C/C++ 解析器；只转发已启用 C/C++ 后的未知语句即可满足第一版。

## 2026-06-27 — 按钮根据大小自动分配字号

### 经验

1. **生成期计算字号**: 按钮尺寸和文本内容在 DSL 解析后已确定，直接在生成 C++ 时计算字号最简单。
2. **owner-draw 字体应用**: 已使用 `BS_OWNERDRAW` 绘制按钮，添加 `HFONT` 并在 `WM_DRAWITEM` 中 `SelectObject` 即可应用字号。

### 教训

1. **小按钮需要下限**: 示例按钮高度只有 10，字号计算必须设置最小值，避免生成不可读或非法字体。
2. **GDI 对象要释放**: `CreateFontA` 创建的字体需要在 `WM_DESTROY` 中 `DeleteObject`。

## 2026-06-27 — 支持命名按钮、点击函数和带空格属性名

### 经验

1. **新增语法从示例反推**: `new button of parent=name`、`.click=click()`、`fn click(){...}` 和 `object.'color of button'=...` 可以在现有最小解析器上增量支持。
2. **点击动作不应提前应用**: 函数体里的属性修改应生成到 `WM_COMMAND`，保持初始 UI 属性不变，点击后再更新并重绘。
3. **带空格属性名用单引号**: 单引号属性名适合保留 `color of button` 这类复合属性，不需要引入更复杂的表达式语法。

### 教训

1. **验证要看 generated cpp**: 示例能编译不代表语义正确，需要检查生成代码确认初始值和点击后的更新逻辑分离。
2. **PowerShell 5 不支持 `&&`**: 顺序验证命令需要分开执行，不能依赖新版 PowerShell 语法。

## 2026-06-27 — 调整按钮文字居中和字号

### 经验

1. **手动测量更稳定**: owner-draw 按钮中使用 `GetTextExtentPoint32A` 获取文字尺寸，再用 `TextOutA` 绘制，比单纯依赖 `DrawTextA` 居中标志更直观。
2. **字号比例要保守**: 按钮较大时，使用高度的 2/5 作为字号上限比 3/5 更适合当前视觉效果。

### 教训

1. **示例尺寸会变化**: 验证时要看当前 `.lc` 示例实际按钮大小，不能沿用旧的 25x10 假设。
2. **生成代码检查很重要**: 需要确认最终 generated cpp 中的 `CreateFontA` 参数和绘制逻辑符合预期。

## 2026-06-27 — 支持按钮 corner 语法

### 经验

1. **属性扩展保持最小闭环**: `.corner=circle|square` 只需扩展 `ButtonSpec`、属性解析和 owner-draw 分支即可支持。
2. **圆角绘制用 GDI 基元**: Win32 owner-draw 中可用 `CreateRoundRectRgn` 裁剪并用 `RoundRect` 绘制圆角按钮。
3. **默认值保持兼容**: 将 `corner` 默认设为 `square`，旧示例和未声明按钮仍走矩形绘制。

### 教训

1. **生成器循环要显式取当前按钮**: 按按钮属性生成不同绘制代码时，每个循环都需要 `const ButtonSpec& button`，避免引用不存在变量。
2. **GDI 对象要成对释放**: `HRGN`、`HBRUSH`、`HPEN` 都要在生成代码中恢复旧对象并释放。

## 2026-06-27 — 支持列表和列表子组件语法

### 经验

1. **列表可先作为容器控件实现**: `new list of parent=name` 可先映射为 Win32 `STATIC` 容器，再把列表子按钮挂到该容器下。
2. **通配父对象语义清晰**: 列表块内 `new button of * = name` 表示父对象为当前列表，适合在解析列表块时直接改写为列表名。
3. **列表外追加子组件可复用父级规则**: `new button of firstlist=name` 可和普通按钮共用解析流程，只在父对象校验和生成父句柄时识别 list。

### 教训

1. **嵌套块不能用普通属性块解析**: list 块里既有 `.x/.y/.aw/.ah`，又有 `new button` 子块，需要专门的 list 块解析流程。
2. **布局应在语义检查后统一计算**: 列表内和列表外子组件都应按同一列表顺序布局，避免只处理列表内的组件。

## 2026-06-27 — 诊断输出 exe 被占用导致的链接失败

### 经验

1. **换输出名可快速定位**: `main.exe` 链接失败但 `main_test.exe` 成功，说明生成代码无误，问题是目标 exe 被占用。
2. **错误提示要区分原因**: 当输出文件已存在且目标编译失败时，提示“exe 正在运行或被占用”比泛泛提示 gcc/g++ 失败更有用。

### 教训

1. **Windows 无法覆盖运行中的 exe**: GUI 程序运行时，链接器覆盖同名输出会出现 `Permission denied`。
2. **调试时可能忘记关闭窗口**: 编译前应关闭上一次运行的 `main.exe`，或临时使用不同 `-o` 输出路径。

## 2026-06-27 — 修复列表子组件不显示

### 经验

1. **owner-draw 消息发给父窗口**: Win32 `BS_OWNERDRAW` 子按钮的 `WM_DRAWITEM` 发给直接父窗口，挂到 `STATIC` 列表容器时主窗口收不到绘制消息。
2. **先用绝对坐标恢复显示**: 将列表子按钮挂到主窗口，并把列表 `.x/.y` 作为偏移量，可以保持现有主窗口 `WndProc` 绘制逻辑不变。

### 教训

1. **容器不是透明布局对象**: 把 list 映射为真实 Win32 父控件会改变消息路由，不能只考虑坐标层级。
2. **生成代码要检查父句柄**: 子组件不显示时，重点检查 `CreateWindowExA` 的 parent 参数和坐标是否符合预期。

## 2026-06-27 — 重写 LimdC UI 测试示例

### 经验

1. **示例要覆盖语法也要可读**: 用一个仪表盘风格示例同时覆盖窗口、圆角按钮、方角按钮、列表、列表内组件和列表外追加组件，比堆叠超大控件更适合作为测试程序。
2. **窗口尺寸应贴近屏幕**: 示例窗口使用 900x620 更容易预览，避免 3300x2500 这类尺寸影响判断。
3. **统一配色更利于验证**: 选择深色底、蓝绿橙强调色，可以更直观看出按钮背景色、文字色和点击变色是否生效。

### 教训

1. **写入反斜杠注释要小心**: 生成 `.lc` 示例时单个 `\` 不会被当前预处理器当成注释，容易导致 `fn main` 识别失败；应优先使用 `//`。
2. **示例编译要用独立输出名**: 使用 `dashboard_demo.exe` 可避免覆盖正在运行的 `main.exe`。

## 2026-06-27 — 简化为列表专项测试示例

### 经验

1. **专项示例更容易排查**: 只保留一个 list、两个 `new button of *` 和一个 `new button of testList`，能直接验证列表布局与子组件显示。
2. **生成代码坐标可作为验收依据**: 检查 generated cpp 中按钮坐标为 `60,50`、`60,122`、`60,194`，能确认列表 `.x/.y/.ah` 偏移计算正确。

### 教训

1. **复杂界面会干扰定位**: 当目标是测试列表功能时，保留太多普通按钮会让问题判断变慢。
2. **注释格式保持一致**: 示例统一使用 `//` 注释，避免单反斜杠被当作源码内容。

## 2026-06-27 — 支持列表标题语法

### 经验

1. **列表标题应影响布局偏移**: `.title` 不只是显示文字，还需要为列表子项预留标题高度，避免标题和第一个子组件重叠。
2. **生成代码可直接验证布局**: 检查 `main.generated.cpp` 中标题 `STATIC` 和子按钮坐标，能确认 `titleHeight` 是否生效。
3. **示例配色要拉开层次**: 使用深色按钮和浅色窗口背景，可以更清楚地区分列表区域、标题和子组件。

### 教训

1. **扫描 list 属性要同步更新**: 新增 `.title` 时，预扫描函数 `isListPropertyLine` 也要同步支持，否则内部按钮尺寸扫描会提前停止。
2. **列表容器和标题都是 Win32 控件**: 添加标题字体后，要在 `WM_DESTROY` 中释放对应 `HFONT`。

## 2026-06-27 — 支持列表边框语法

### 经验

1. **带空格属性可通过属性名正则支持**: `.frame color=#000000` 可以解析为属性名 `frame color`，不需要强制用户加引号。
2. **自定义边框颜色应手绘**: Win32 `WS_BORDER` 不能指定颜色，改为在主窗口 `WM_PAINT` 中用 `CreatePen` + `Rectangle` 绘制。
3. **布尔属性要单独校验**: `.frame=true/false` 使用 `parseBoolean`，错误提示比把它当字符串更明确。

### 教训

1. **新增 list 属性要更新三处**: `ListSpec` 字段、list 属性预扫描、`makeList` 属性分派都要同步修改。
2. **不要混用默认边框和自绘边框**: `WS_BORDER` 会产生系统默认边框颜色，启用自定义边框时应移除，避免视觉冲突。

## 2026-06-27 — 修复列表四边边框显示

### 经验

1. **四边边框要显式画四条线**: 用 `MoveToEx` + 连续 `LineTo` 比 `Rectangle` 更容易确认上、右、下、左都生成了绘制代码。
2. **边框范围要略大于子控件**: 在列表宽高基础上加少量像素，能避免右边和底边贴住按钮边缘而被视觉上吞掉。
3. **显示后触发重绘**: 窗口显示和子控件创建后调用 `RedrawWindow`，能让边框在初次显示时更稳定出现。

### 教训

1. **父窗口绘制可能被子控件遮挡**: 列表子按钮是同级子控件时，边框如果和控件完全重合，容易看起来缺边。
2. **验证时注意 exe 占用**: 如果输出 exe 正在运行，链接会失败；应关闭窗口或换新输出名。

## 2026-06-27 — 优化列表卡片风格

### 经验

1. **行动前让用户选择方向**: 对视觉优化先给风格、改动范围和新语法选项，能避免直接改偏。
2. **卡片风格需要背景和内边距**: 仅有边框会显得生硬，新增 `.background color` 和 `.padding` 后列表区域更清晰。
3. **标题颜色单独控制**: `.title color` 让标题能和按钮区分层级，适合作为 list 的独立视觉属性。
4. **STATIC 颜色要处理消息**: Win32 标题和列表背景需要通过 `WM_CTLCOLORSTATIC` 返回画刷并设置文字颜色。

### 教训

1. **生成器插入新消息分支要保持原有 WM_DRAWITEM**: 插入 `WM_CTLCOLORSTATIC` 时不能覆盖按钮 owner-draw 分支。
2. **GDI 画刷需要释放**: 为列表背景创建的 `HBRUSH` 要在 `WM_DESTROY` 中删除。

## 2026-06-27 — 支持 text 组件语法

### 经验

1. **text 组件可以先映射为 STATIC**: `new text=name{...}` 用 Win32 `STATIC` 控件即可支持显示文本、背景色、文字色和字体大小。
2. **表达式语法要转成 C++ 表达式**: `.text=printf(txt)` 和 `.text=cout<<txt` 都可归一为变量表达式 `txt`，再用 `LimdCToString` 转字符串。
3. **原生 C++ 变量需要命名空间支持**: 示例中的 `string txt` 需要生成 `using namespace std;`，否则目标代码无法编译。

### 教训

1. **新增 UI 控件要同步资源释放**: text 的 `HFONT` 和背景 `HBRUSH` 都需要在 `WM_DESTROY` 删除。
2. **文本控件颜色依赖 WM_CTLCOLORSTATIC**: 仅创建 STATIC 不够，背景色和文字色要在 `WM_CTLCOLORSTATIC` 返回对应画刷。

## 2026-06-27 — 支持 text center 对齐语法

### 经验

1. **默认行为要符合 DSL 语义**: text 默认应从左上角开始显示，只有 `.center=1` 才生成居中样式。
2. **布尔数字属性要校验范围**: `.center` 只接受 `0` 或 `1`，错误值应在编译期报错。

### 教训

1. **Win32 STATIC 默认样式要显式控制**: 如果不去掉 `SS_CENTER`，即使 DSL 写 `.center=0` 也会继续居中。
2. **验证生成代码比只看 exe 更直接**: 检查生成的 `CreateWindowExA` 样式能快速确认对齐逻辑。
